# Agent 架构说明

## 1. 总览

本系统采用**多 Agent 共享内存架构**，由 5 个专职 Agent 组成，通过 Python 模块调用实现零延迟通信。

```
┌─────────────────────────────────────────────────┐
│              React + ECharts 前端                 │
└──────────────────────┬──────────────────────────┘
                       │ HTTP API (FastAPI)
┌──────────────────────▼──────────────────────────┐
│                FastAPI 调度层                     │
│    接收请求 → 调度 Agent → 返回结果               │
└──┬──────┬──────┬──────┬──────┬──────────────────┘
   │      │      │      │      │
┌──▼───┐┌─▼────┐┌─▼────┐┌─▼───┐┌─▼────┐
│Parser││Extract││Integr││ RAG ││ Chat │
│Agent ││Agent  ││Agent ││Agent││Agent │
└──────┘└───────┘└──────┘└─────┘└──────┘
         共享内存（Python 模块调用）
```

### 为什么选多 Agent 而非单 Agent

| 维度 | 单 Agent | 多 Agent |
|---|---|---|
| 职责分离 | 所有逻辑混在一起 | 每个 Agent 职责单一 |
| 可调试性 | 出错难以定位 | 精确到具体 Agent |
| 代码质量 | 大文件难维护 | 模块化，结构清晰 |
| 架构文档 | 设计决策少 | 每个 Agent 有独立设计论证 |
| 通信开销 | 无 | Python 模块调用 ≈ 0 |

### 为什么用共享内存而非消息队列

- 5 小时黑客松时间约束，无需分布式通信的可靠性保证
- Python 模块调用延迟 < 1ms，远优于 HTTP/gRPC
- 代码量更少，调试更简单

## 2. Agent 详解

### 2.1 ParserAgent — 教材解析

**文件**: `backend/services/parser_agent.py`

**职责**: 将 PDF 教材解析为统一的结构化数据（TextbookInfo）

**核心算法**:
1. **逐页 Markdown 转换**: 使用 pymupdf4llm 将每页转为 Markdown，保留标题层级
2. **页眉页脚过滤**: 统计每页首尾 3 行出现频率，>50% 页面出现的行判定为页眉/页脚并移除
3. **图表标注过滤**: 连续 3 行以上、每行 < 15 字的文本判定为图表内部标注（坐标轴标签、图例），跳过
4. **章节检测**: 正则匹配 `^第X章` 模式，将全文按章分割
5. **表格标题保留**: 正则提取 `表X-X 标题` 格式的表格描述

**输出**: `TextbookInfo` — 章节列表 + 内容 + 元数据

### 2.2 ExtractorAgent — 知识提取

**文件**: `backend/services/extractor_agent.py`

**职责**: 从教材章节中提取知识点和关系，构建知识图谱

**三级递进提取策略**:

```
Level 1: 逐节提取（每个小节独立调 LLM）
  → 上下文：当前小节内容（~2000字）
  → 产出：10-15 个知识点 + 节内关系

Level 2: 跨节关系补充（滑动窗口，每 3 节一组）
  → 上下文：30-45 个节点名称
  → 产出：跨节关系边

Level 3: 按章补充远距离关系（每章独立处理）
  → 上下文：本章节点 vs 其他章节点名称
  → 产出：跨章远距离关系边
```

**为什么分三级**:
- Level 1 粒度最细，但看不到节间关系
- Level 2 补充相邻节的关系，窗口大小适中（30-45 节点名 ≈ 500 字）
- Level 3 补充远距离关系，按章分批避免上下文溢出（全书 ~500 节点名会超出 LLM 窗口）

**关系类型**:
- `prerequisite`: 前置依赖（学习 B 前必须掌握 A）
- `parallel`: 并列关系（同一层级的平行概念）
- `contains`: 包含关系（上位概念与下位概念）
- `applies_to`: 应用关系（某知识点是另一个的应用场景）

### 2.3 IntegratorAgent — 跨教材整合

**文件**: `backend/services/integrator_agent.py`

**职责**: 将多本教材的知识图谱合并，识别重叠知识点

**两阶段对齐算法**:

```
阶段 1: Embedding 粗筛（快，零 LLM 成本）
  → 每个知识点的 "名称+定义" 向量化
  → 不做 category 预过滤（医学分类边界模糊）
  → 余弦相似度 ≥ 0.70 → 候选合并对

阶段 2: LLM 精确判断（慢，但只处理候选对）
  → 每批 5 个候选对，LLM 判断 merge/keep
  → 输出决策 + 理由 + 置信度
```

**为什么不按 category 预过滤**:
- "炎症" 在病理学中是"病理变化"，在生理学中是"生理过程"
- 预过滤会漏掉这类跨类别匹配

**压缩比控制**:
- 目标: 整合后总字数 ≤ 原始总字数的 30%
- merge 操作合并重复内容，remove 操作删除冗余节点

**前置依赖链保护**:
- 每次 merge/remove 前检查是否影响 prerequisite 关系
- 被合并的节点的依赖关系转移到保留节点上

### 2.4 RAGAgent — 精准问答

**文件**: `backend/services/rag_agent.py`

**职责**: 基于整合后的知识库进行检索增强生成问答

**Pipeline**:
```
文档分块（600字/块，80字重叠）
    ↓
向量嵌入（bge-small-zh-v1.5，512维）
    ↓
ChromaDB 存储（~5500 chunks，~11MB）
    ↓
用户提问 → 向量化 → 检索 top-5
    ↓
LLM 生成回答 + 来源引用
```

**为什么 600 字/chunk**:
- bge-small-zh-v1.5 有效上下文约 512 token（中文约 300-400 字）
- 600 字在截断后仍保留完整语义段落

**为什么 80 字重叠**:
- 确保跨段落概念不因分块丢失

### 2.5 ChatAgent — 多轮对话

**文件**: `backend/services/chat_agent.py`

**职责**: 用户意图→系统行为的翻译层

**意图识别**: 关键词匹配 + LLM 兜底

| 意图 | 关键词 | 动作 |
|---|---|---|
| 解释合并 | "为什么" + "合并" | 查询决策日志 |
| 解释删除 | "为什么" + "删除" | 查询决策日志 |
| 保留节点 | "保留" / "恢复" | 修改决策为 keep |
| 删除节点 | "删除" | 新增 remove 决策 |
| 拆分节点 | "分开" / "拆分" | 修改合并决策为 keep |
| 手动合并 | "合并" | 新增 merge 决策 |
| 查询状态 | "状态" / "统计" | 返回决策统计 |

## 3. 数据流

```
PDF 上传 → ParserAgent → TextbookInfo
                              ↓
                     ExtractorAgent → KnowledgeGraph
                              ↓
                     RAGAgent → ChromaDB 索引
                              ↓
（重复处理所有教材）
                              ↓
                     IntegratorAgent → MergedKnowledgeGraph + 决策
                              ↓
                     前端展示（图谱 + 决策列表 + RAG + 对话）
```

## 4. 设计取舍

| 取舍 | 选择 | 原因 |
|---|---|---|
| 前端框架 | React + shadcn/ui | Gradio 无法实现 ECharts 力导向图的交互需求 |
| Agent 通信 | 共享内存 | 5 小时内无需分布式可靠性 |
| Embedding 模型 | bge-small-zh-v1.5 | 本地运行，无 API 成本，中文效果好 |
| 意图识别 | 关键词 + LLM 兜底 | 5 小时内实现足够，覆盖高频意图 |
| 图谱可视化 | ECharts graph | 成熟的力导向图实现，支持缩放/拖拽/点击 |

## 5. 创新点

1. **三级递进知识提取**: 从节内→跨节→跨章逐步扩大视野，平衡 LLM 上下文限制与关系完整性
2. **ChatAgent 控制平面**: 将对话作为修改整合方案的接口，每个赛题要求映射到具体 Agent API
3. **Embedding 粗筛 + LLM 精确判断**: 两阶段对齐平衡效率与准确度
4. **无 category 预过滤**: 适应医学领域分类边界模糊的特点
