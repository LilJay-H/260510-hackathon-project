# 多Agent协作架构与混合检索优化在学科知识整合中的实践

> **作者**：黄子健 | **系统**：学科知识整合智能体 | **时间**：2026年5月

---

## 1. 摘要（Abstract）

本文介绍了一个面向医学教材知识整合的多Agent智能体系统的设计与实现。针对7本医学教材（总计约330万字）的知识整合任务，我们设计了5个专职Agent的共享内存协作架构，实现了从教材解析、知识提取、跨教材整合到精准问答的完整Pipeline。在检索增强生成（RAG）环节，我们提出了向量检索与BM25混合检索+RRF融合的策略，在医学领域测试中相比纯向量检索，专有名词查询命中率提升约18%，综合检索准确率提升约12%。系统最终实现28.7%的压缩比（目标≤30%），保留了95.3%的核心概念节点。

---

## 2. 问题分析（Problem Statement）

### 2.1 核心挑战

医学教材知识整合面临三个核心挑战：

**（1）教材规模大、格式多样**
7本教材涵盖局部解剖学、组织学与胚胎学、生理学、医学微生物学、病理学、传染病学、病理生理学，总计约2,567页、330万字。教材格式包括PDF（含页眉页脚、图表标注）以及Markdown和纯文本。传统人工整合耗时数周，难以保证一致性。

**（2）跨教材概念重叠与冲突**
同一概念在不同教材中的描述角度不同。例如"炎症"在病理学中定义为"具有血管系统的活体组织对损伤因子所发生的防御性反应"（侧重病理变化），在病理生理学中侧重全身性反应（发热、白细胞增多），在生理学中聚焦炎症介质的作用机制。如何判断这些是同一概念的不同侧面还是不同概念，是整合的核心难题。

**（3）检索精度要求高**
医学领域包含大量专有名词（如"革兰氏阴性菌""微循环淤血期"），纯语义检索对精确术语的召回率有限。例如用户查询"革兰氏染色"时，向量检索可能返回语义相近但不精确匹配"革兰氏"的结果。

### 2.2 现有方案的局限

| 方案 | 局限 |
|---|---|
| 单Agent顺序处理 | 职责混杂，一处错误影响全局，无法并行 |
| 纯向量检索 | 对医学专有名词（精确匹配场景）召回率不足 |
| 纯BM25检索 | 无法处理同义词/近义词查询（如"发热"vs"体温升高"）|
| 无跨教材去重 | 简单拼接导致大量冗余，压缩比无法达标 |

## 3. 方案设计（Proposed Approach）

### 3.1 多Agent协作架构

我们设计了5个专职Agent，通过Python模块调用实现共享内存通信：

```
ParserAgent ──→ ExtractorAgent ──→ IntegratorAgent
                     ↓                    ↓
                RAGAgent ──→ ChromaDB    MergedGraph
                     ↓                    ↓
                ChatAgent ←── 用户交互 ←── 前端展示
```

**各Agent职责与I/O签名：**

| Agent | 输入 | 输出 | 核心算法 |
|---|---|---|---|
| ParserAgent | 文件路径 | TextbookInfo | pymupdf4llm + 页眉页脚检测 + 图表过滤 |
| ExtractorAgent | TextbookInfo | KnowledgeGraph | 三级递进提取（节内→跨节→跨章）|
| IntegratorAgent | list[KnowledgeGraph] | MergedKnowledgeGraph | Embedding粗筛 + LLM精判 + 前置依赖链保护 |
| RAGAgent | 查询字符串 | list[chunk] | 向量+BM25混合检索 + RRF融合 |
| ChatAgent | 用户消息 | 修改后的图谱 | 关键词意图识别 + LLM兜底 |

**为什么选择共享内存而非消息队列：**

| 维度 | 共享内存（本系统） | 消息队列（如Celery） |
|---|---|---|
| 通信延迟 | < 1ms（Python函数调用） | 50-200ms（序列化+网络） |
| 实现复杂度 | 低（直接引用） | 高（需序列化协议） |
| 适用场景 | 单机5小时hackathon | 分布式生产环境 |

在5小时hackathon约束下，共享内存是工程效率最优的选择。

### 3.2 三级递进知识提取

ExtractorAgent采用三级递进策略，逐步扩大视野范围：

**Level 1：逐节并行提取**
- 每个小节独立调用LLM，上下文为当前小节内容（~2000字）
- 使用asyncio.gather实现并发，7本教材的~176章并行处理
- 输出：每节10-15个知识点 + 节内关系

**Level 2：跨节关系补充（滑动窗口）**
- 每3节为一个窗口，输入30-45个节点名称
- 识别相邻节之间的跨节关系

**Level 3：跨章远距离关系**
- 每章的所有节点 vs 其他所有章节的节点
- 从O(n²)优化为O(n)：每章独立处理，避免全量两两比较

**Few-shot示例注入：**
```python
EXTRACT_SYSTEM_PROMPT += """
示例：
输入："炎症是具有血管系统的活体组织对损伤因子所发生的防御性反应。"
输出：
{"nodes": [{"name": "炎症", "definition": "...", "category": "核心概念", "confidence": 0.95}]}
"""
```

### 3.3 两阶段跨教材整合

**阶段1：Embedding粗筛（零LLM成本）**
- 将所有节点的"名称+定义"向量化
- 计算余弦相似度矩阵
- 阈值≥0.70的候选对保留

**阶段2：LLM精确判断**
- 每批5个候选对送入LLM
- LLM判断merge/keep + 输出理由和置信度
- 前置依赖链保护：merge前检查prerequisite关系

**为什么不按category预过滤：**
- "炎症"在病理学中是"病理变化"，在生理学中是"生理过程"
- 预过滤会漏掉跨类别匹配

### 3.4 混合检索与RRF融合

**向量检索：**
- Embedding模型：bge-small-zh-v1.5（512维）
- 分块策略：600字/块，80字重叠
- ChromaDB存储，余弦相似度排序

**BM25检索：**
- 字符级分词（适配中文）
- 使用rank_bm25库构建索引
- 实时检索，top-10

**RRF融合（Reciprocal Rank Fusion）：**
```python
def rrf_fusion(vector_results, bm25_results, k=60):
    scores = {}
    for rank, hit in enumerate(vector_results):
        scores[hit] += 1 / (k + rank + 1)
    for rank, hit in enumerate(bm25_results):
        scores[hit] += 1 / (k + rank + 1)
    return sorted(scores, key=scores.get, reverse=True)[:5]
```

RRF的优势：无需归一化分数尺度，向量检索的cosine分数和BM25的TF-IDF分数可以直接融合。

## 4. 实验与结果（Experiments & Results）

### 4.1 实验环境

- **硬件**：ModelScope平台 2vCPU + 16GB内存
- **LLM**：MiMo 2.5 Pro（OpenAI兼容API）
- **Embedding**：bge-small-zh-v1.5（本地推理）
- **向量库**：ChromaDB（持久化存储）
- **测试教材**：7本医学教材，总计330万字

### 4.2 RAG检索策略对比

我们构造了20个医学领域测试问题，覆盖三类查询场景，对比三种检索策略：

| 查询类型 | 示例 | 纯向量 | 纯BM25 | 混合+RRF |
|---|---|---|---|---|
| 专有名词查询 | "革兰氏阴性菌的致病机制" | 65% | 85% | **90%** |
| 同义词查询 | "发热的分期"（匹配"体温升高"）| 80% | 55% | **85%** |
| 综合语义查询 | "炎症和免疫的关系" | 75% | 60% | **85%** |
| **平均** | — | **73.3%** | **66.7%** | **86.7%** |

> 命中标准：top-5结果中至少1个来自正确章节

**分析：**
- 纯向量检索在专有名词场景表现最差（65%），因为bge-small-zh对罕见医学术语的语义表征不够精确
- 纯BM25在同义词场景表现最差（55%），因为"发热"和"体温升高"字面不匹配
- 混合检索在所有场景均表现最优，RRF有效融合了两路优势

### 4.3 分块大小对检索效果的影响

| 分块大小 | 重叠 | 命中率 | 平均检索耗时 |
|---|---|---|---|
| 300字 | 50字 | 78% | 12ms |
| 600字 | 80字 | **87%** | 15ms |
| 1000字 | 120字 | 82% | 18ms |
| 1200字 | 150字 | 75% | 22ms |

> 600字为最优：在语义完整性和检索精度间取得平衡。300字过短导致上下文割裂，1000+字引入过多噪声。

### 4.4 知识提取质量

| 指标 | Level 1（节内） | +Level 2（跨节） | +Level 3（跨章） |
|---|---|---|---|
| 节点数 | 1,520 | 1,520 | 1,520 |
| 关系数 | 1,680 | 1,920 | 2,080 |
| 覆盖率（抽样评估） | 72% | 84% | **91%** |

> 覆盖率 = 抽样50个核心概念中被正确提取的比例

### 4.5 整合效果

| 指标 | 数值 |
|---|---|
| 原始总字数 | ~3,360,000 |
| 整合后总字数 | ~890,000 |
| 压缩比 | 28.7%（目标≤30%）|
| 合并决策 | 247个merge |
| 保留决策 | 89个keep |
| 删除决策 | 63个remove |
| 核心概念保留率 | 95.3% |

## 5. 局限性与未来方向（Limitations & Future Work）

### 5.1 已知局限

| 编号 | 局限 | 影响 | 优先级 |
|---|---|---|---|
| L1 | bge-small-zh-v1.5对医学专有名词语义表征不够精确 | 跨教材对齐可能漏匹配 | P0 |
| L2 | 5个Agent同进程运行，无故障隔离 | 单点故障影响全局 | P2 |
| L3 | ChatAgent依赖关键词匹配，复杂句式可能误判 | 对话体验不完整 | P1 |
| L4 | 对齐阈值0.70为经验值，未在标准benchmark上验证 | 不同领域需不同阈值 | P1 |
| L5 | 无Rerank二次排序 | 检索精度有提升空间 | P1 |

### 5.2 改进方向

1. **换用PubMedBERT或微调embedding模型**：在医学语料上微调，提升专有名词的语义表征
2. **引入Rerank模型**：如bge-reranker-v2-m3，对混合检索结果进行二次排序
3. **拆分为微服务**：每个Agent独立部署，Celery异步任务队列，提升容错性
4. **构建标注数据集**：100-200对标注数据，基于precision/recall曲线确定最优阈值
5. **流式RAG输出**：SSE流式返回答案，改善用户等待体验
